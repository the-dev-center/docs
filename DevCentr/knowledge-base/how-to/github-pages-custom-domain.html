GitHub Pages: Use Your Own Domain

This guide tells you how to make your GitHub Pages site open at your own domain instead of yourname.github.io. We use short sentences and explain every step. If you get stuck, the official GitHub doc has more detail.

What do I need? (Quick list)

You do two things: (1) tell GitHub your domain name, and (2) tell the place that owns your domain (your "DNS provider") where to send visitors. The table below shows what to add at your DNS provider depending on what you want.

  • Apex domain = your main domain with no extra part in front. Example: example.com.

  • Subdomain = something in front of your domain. Examples: www.example.com, blog.example.com. For GitHub Pages, www is the one people use most.

  • DNS provider = the place where you manage your domain (where you bought it or where you set its "DNS records"). Examples: Cloudflare, Namecheap, Google Domains, your registrar.

Order of work

  1. First: Add your custom domain in GitHub (Settings → Pages → Custom domain). Save. That way GitHub knows you own it and no one else can use it for their site.

  2. Second: Add the DNS records at your DNS provider (see the table and sections below). If you do DNS first and forget GitHub, someone else could point your domain to their site.

DNS changes can take up to 24 hours to show up. Be patient.

Step 1: Add the domain in GitHub

  1. Open your repository on GitHub.

  2. Click Settings.

  3. In the left sidebar, under "Code and automation", click Pages.

  4. Find the Custom domain box.

  5. Type your domain (e.g. example.com or www.example.com). Do not type https:// or a path.

  6. Click Save.

If you publish from a branch, GitHub will add a file called CNAME to your repo. If you build the site with GitHub Actions, you do not need that file.

If you deploy with GitHub Actions (e.g. upload-pages-artifact + deploy-pages): In Settings → Pages, under "Build and deployment", set Source to GitHub Actions. If Source is "Deploy from a branch", GitHub will ignore the Actions deployment and your custom domain may not serve the built site.

After the domain is set, you can turn on Enforce HTTPS when it appears (sometimes it takes a while). That makes your site load over a secure connection.

Step 2a: You want the main domain only (e.g. example.com)

This is called an apex or root domain. You need either four A records or one ALIAS/ANAME record at your DNS provider.

Go to your DNS provider and create four A records. Each record points your domain to one of these four numbers (IP addresses):

  • Name / host: Use @ or your apex (e.g. leave blank or example.com depending on the provider).

  • TTL: Default is fine (e.g. 3600 or "Auto").

    If you use Cloudflare: each record has a proxy toggle (cloud icon). Use grey cloud (DNS only) for these records. Orange cloud (proxied) can leave "Enforce HTTPS" unavailable or cause certificate errors; grey avoids that. If you later try orange, set SSL/TLS to Full or Full (strict) in Cloudflare.

Option B: One ALIAS or ANAME record (if your DNS provider has it)

Some providers (e.g. Cloudflare, DNSimple) let you use one record instead of four:

  • Type: ALIAS or ANAME

  • Name / host: @ or your apex

  • Value / target: yourusername.github.io or yourorg.github.io (user or org site). No repo name. No https://.

    If you use Cloudflare: set this record to grey cloud (DNS only). Proxied (orange) can break Enforce HTTPS or cause certificate issues.

If you do not see ALIAS or ANAME in your provider, use the four A records above.

Optional: IPv6 (AAAA records)

If you want IPv6 support, add four AAAA records. Name is the same as your A records. Values:

  • 2606:50c0:8000::153

  • 2606:50c0:8001::153

  • 2606:50c0:8002::153

  • 2606:50c0:8003::153

Step 2b: You want www only (e.g. www.example.com)

This is a subdomain. You need one CNAME record.

  1. Go to your DNS provider.

  2. Add a CNAME record:

    • Type: CNAME

    • Name / host: www (or www.example.com if the provider asks for the full name)

    • Value / target: yourusername.github.io or yourorg.github.io

  3. Do not put the repository name in the value. Do not put https://. Just the .github.io address.

    If you use Cloudflare: set this CNAME to grey cloud (DNS only). Orange cloud (proxied) often keeps "Enforce HTTPS" greyed out or causes certificate errors.

Step 2c: You want both the main domain and www

  1. Do "Step 2a" for your main domain (four A records or one ALIAS/ANAME). If you use Cloudflare, use grey cloud (DNS only) for those records.

  2. Do "Step 2b" for www (one CNAME pointing to yourusername.github.io or yourorg.github.io). Same Cloudflare rule: grey cloud for the CNAME.

  3. In GitHub (Settings → Pages → Custom domain), type one of the two (either example.com or www.example.com) and save. GitHub will then redirect the other one to the one you chose. For example, if you set example.com in GitHub, then www.example.com will redirect to example.com.

Check that it worked

After you save your DNS records, wait a few minutes (or up to 24 hours). Then:

  • Windows (PowerShell): Run Resolve-DnsName example.com (use your domain instead of example.com). You should see the GitHub IPs or a CNAME to .github.io.

  • Mac / Linux: Run dig example.com +short (use your domain). You should see the four IPs or a CNAME to your .github.io address.

If you see the right numbers or the right CNAME, your DNS is set. Your site should load at your domain once GitHub has finished updating (and after you turned on Enforce HTTPS if you want https).

Remove a custom domain

If you want to stop using your domain and go back to yourname.github.io:

  1. Open the repo on GitHub → Settings → Pages.

  2. Under Custom domain, click Remove.

Then you can remove or change the DNS records at your DNS provider so the domain points somewhere else (or nowhere).

Safety tips

  • Verify your domain in GitHub before or right after adding it. That helps stop someone else from using your domain on their repo. See GitHub’s doc on "Verifying your custom domain."

  • Do not use wildcards like *.example.com in DNS. That can let other people use subdomains of your domain (e.g. something.example.com) for their site. Use exact names only (e.g. www, or your apex).

  • If you turn off GitHub Pages but leave the domain in DNS pointing at GitHub, someone else could use your domain. If you disable the site, remove or change the DNS records too.

Troubleshooting: “Works on .github.io but not my domain” or “GitHub Actions deploy but domain is wrong”

If your workflow uses actions/upload-pages-artifact and actions/deploy-pages:

  1. In the repo on GitHub go to Settings → Pages.

  2. Under Build and deployment, check Source. It must say GitHub Actions. If it says Deploy from a branch, GitHub is not using your Actions workflow for this site; change it to GitHub Actions and save.

  3. Under Custom domain, make sure your domain (e.g. devcentr.org or www.devcentr.org) is set and saved. GitHub does not read the custom domain from a CNAME file when you use GitHub Actions.

If the domain still fails, run Resolve-DnsName yourdomain.com -Type A (PowerShell) or dig yourdomain.com +short (Mac/Linux). For an apex domain you should see the four GitHub Pages IPs (see "Step 2a" above). For www you should see a CNAME to yourorg.github.io.

Summary

I want…

I do this

My site at example.com (no www)

Add domain in GitHub. At DNS: four A records (or one ALIAS) with the four IPs (or ALIAS to .github.io).

My site at www.example.com

Add domain in GitHub. At DNS: one CNAME — name www, value yourusername.github.io (or org).

My site at both example.com and www.example.com

Add domain in GitHub (use one of the two). At DNS: A (or ALIAS) for apex + CNAME for www. GitHub does the redirect.
