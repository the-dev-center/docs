Package Registries: Where to Publish?

Once you have a release, where should you publish it? The answer depends on your language, package type (library vs. CLI vs. container), and target audience.

The \"GitHub Packages\" Misconception

GitHub Packages supports specific ecosystems: npm, NuGet, Maven, Gradle, RubyGems, and Docker.

It does not verify or host \"generic\" software.

For a project in a language that is not one of those ecosystems (e.g. D, Go, Rust, Zig):

  • Is it advisable to publish to GitHub Packages (npm/NuGet) just to be on GitHub?

  • No. Wrapping a native binary in a Node.js or .NET shim adds complexity for no benefit unless your primary users are strictly JS/.NET developers.

  • Exception — Docker (ghcr.io): Publishing a container image to GitHub Container Registry is often useful. It makes your tool language-agnostic: other projects can use it in CI without installing your language runtime. Example: docker run --rm -v $(pwd):/src ghcr.io/your-org/your-tool:latest

Choosing Registries: A General Approach

Use this checklist for any software project:

  1. Binaries / executables: Publish to GitHub Releases (or equivalent). This is the source of truth; other distributors (package managers) typically point to these URLs.

  2. Library / package in an ecosystem: Publish to that ecosystem’s official registry (e.g. npm, PyPI, crates.io, Hex, code.dlang.org) so users can install with their normal tooling.

  3. Containers: Use GitHub Container Registry (or another OCI registry) if you need portability across languages and CI environments.

  4. Skip: Putting native/non-ecosystem software on npm/NuGet/Maven etc. unless you are explicitly targeting that ecosystem.

Registry Examples by Context

Registry type Purpose When to use

Ecosystem registry

Official registry for your language (npm, PyPI, crates.io, Hex, code.dlang.org, etc.)

Use for libraries. Enables pip install, cargo add, dub fetch, etc.

GitHub Releases

Raw binaries and assets

Always. Source of truth for versioned artifacts; other systems reference these URLs.

Winget

Windows Package Manager

Recommended for Windows CLI tools and desktop apps.

Chocolatey

Legacy Windows package manager

Optional for enterprise or older Windows environments.

Homebrew

macOS (and Linux) package manager

Recommended if you support macOS (or Linux) users.

Docker / ghcr.io

Container images

Use when you want language-agnostic, CI-friendly distribution.

Summary

  1. Binaries: Use GitHub Releases (or equivalent) first.

  2. Libraries: Use your language’s official registry so users can install with standard tools.

  3. Containers: Use GitHub Container Registry (or another OCI registry) when you need portability.

  4. Skip forcing non-ecosystem software into npm/NuGet/Maven unless that ecosystem is your target.
